网络服务

Web服务，分布式对象等等，大都有以下基本结构：

    Read request
    Decode request
    Process service
    Encode reply
    Send reply

但他们的本质和每个步骤的代价都存在着差异

可扩展性的目标

  • 在负载增加情况下（更多客户端）优雅降级（体面退化，故障弱化）
  • 在增加资源情况下（CPU，内存，硬盘，带宽）能持续提升性能
  • 同样满足可用性和性能目标
    • 低延迟
    • 满足高峰需求
    • 可调的服务质量
  • 分治通常是最好的方式实现以上的可扩展目标

分而治之

  • 划分处理成细小的任务
     每个任务的执行都非阻塞
  • 执行每个就绪的任务
    在这里，一个IO事件通常作为触发器服务
  • java.nio提供的基本机制
    • 非阻塞（Non-Blocking）读和写
    • 分派（Dispatch）与感知IO事件关联的任务
  • 无尽变化的可能
    事件驱动模式的种类

事件驱动模式

  • 通常比其他方案更高效
    • 较少资源消耗 - 不需要为每个客户端开启一条线程
    • 较少开销 - 更少的上下文切换，通常需要较少的锁
    • 但分派（dispatch）可能更慢 - 必须手动绑定操作逻辑到事件
  • 通常编程难度更大
    • 必须拆分成简单非阻塞操作
      • 和GuI的事件驱动操作相似
      • 不能消除所有的阻塞： GC，page faults等等
    • 必须监控追踪服务的逻辑状态

背景知识：AWT的事件

事件驱动-IO使用相似的思想但不同的模式

Reactor模式

  • Reactor通过分派到合适的Handler来响应IO事件
    • 和AWT线程相似
  • Handler执行非阻塞操作
    • 和AWT的ActionListeners相似
  • 通过绑定handlers到event进行管理
    • 和AWT的addActionListener相似
  • 参见Schmidt et al,Pattern-Oriented Software Architecture, Volume2(POSA2)

基本Reactor设计

java.nio.Support

  • Channels
    连接到文件，socket等等支持非阻塞读操作的
  • Buffers
    类似数组的对象，能够直接通过Channel读写
  • Selectors
    告知哪些Channel集合有IO events
  • SelectionKeys
    维持IO event状态和绑定

多线程设计

  • 策略上添加线程增加可扩展性
    • 主要应用在多处理器
  • 工作线程
    • Reactors应该快速触发Handlers
      • Handler的处理是慢于Reactor
    • 卸下非阻塞（NIO）处理到其他线程
  • 多个Reactor线程
    • Reactor线程能够忙于IO处理
    • 分布负载到其他reactors
      • 负载均衡到相配的cpu和IO比例

工作线程

  • 卸下nio的处理操作，加速了Reactor线程
    • 和POSA2的Proactor设计相似
  • 比起将compute-bound的处理改造成事件驱动形式更简单
    • 应该仍是纯非阻塞的计算
      • 足够的处理能力对应过重的开销
  • 但是更难重叠处理IO
    最好的情形是第一次读数据就能获取所有的输入，写到buffer
  • 使用线程池，使有能力调节和控制
    • 一般只需要比客户端少很多的线程数量

Worker Thread Pools

Handler with Thread Pool

协调任务

  • 切换
    • 每个任务使可行，触发或是调用下一个任务，这通常是最快的，但是架构脆弱
  • 回调 到每个Handler的分派器
    • 设置状态，attachment，etc
    • GoF Mediator模式的变种
  • 队列
    • 例如，通过传递buffer跨过阶段
  • Future
    • 当每个任务处理出结果
    • 除了join或wait/notify外的分层协调

使用线程池

  • 可调的工作线程池
  • 主要方法execute(Runnable r)
  • 可控制以下：
    • 任务队列的类型（任何Channel）
    • 最大的线程数量
    • 最小的线程数量
    • “温和”以及相对的命令式线程
    • 保持存活，直到空闲线程死亡
      • 如果有必要，之后会有新的线程取代
    • 饱和策略
      • 阻塞，丢弃，生产者运行等等

多个Reactor线程

  • 使用Reactor池
  • 使用相配的CPU和IO比例
  • 静态或动态的构造
      每个实例拥有自己的Selector, Thread, dispatch loop等等
  • Main acceptor分布到其他的reactor

    使用多个Reactors

使用java.nio的其他特性

  • 每个Reactor配有多个Selectors
    • 将不同的handler绑定到不同的IO事件
    • 可能协作需要小心地使用同步
  • 文件传输
    • 自动化 file-to-net 或是 net-to-file 复制
  • 内存映射文件
    • 通过buffers访问文件
  • 直接内存Buffer
    • 有时候能实现零拷贝
    • 但是需要设置和终结开销
    • 最好是使用长连接的应用

基于连接的拓展

  • 使用简单的服务请求替代
    • 客户端建立连接
    • 客户端发送一系列messages/requests
    • 客户端断开连接
  • 例子
    • 数据库和事务的监控
    • 多人游戏，聊天等等
  • 能够拓展基本网络服务的模式
    • 处理大量相关的长连接客户端
    • 追踪客户端和session状态（包括丢弃）
    • 通过多个hosts进行分布式服务
